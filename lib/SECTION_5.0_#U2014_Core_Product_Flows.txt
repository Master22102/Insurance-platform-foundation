SECTION 5.0 — Core Product Flows (Canonical, mobile-first)
5.0.0 Constitutional Purpose
The platform is a structured decision-support system for travel before, during, and after trips.
Hard rules:
* Requires at least one anchor: Trip and/or Policy.

* No silent mutation. No silent realignment.

* Voice is authoritative only through proposal → explicit confirmation.

* Identical ItineraryHash returns identical Quick Scan result (account-bound, rate-limited).

Amendment — Draft Home Workflow
March 2026 | Addendum to Section 5.0
________________


Purpose of Amendment
This amendment adds Step 4D (Draft Home workflow) to the core product flow, expanding the trip creation process to include incremental authoring before Deep Scan.
All existing Section 5.0 flows remain intact. This amendment adds a new pre-readiness workflow without modifying existing steps.
4D — Draft Home Workflow (Between Trip Creation & Deep Scan)
Flow Position
Previous flow (Section 5.0):
Step 4A: Account Creation & Onboarding
Step 4B: Attach First Policy (optional)
Step 4C: Create Trip (minimal data)
Step 4E: Run QuickScan OR DeepScan
Updated flow (v1.1):
Step 4A: Account Creation & Onboarding
Step 4B: Attach First Policy (optional)
Step 4C: Create Trip (minimal data) → Trip in DRAFT state
Step 4D: Draft Home Workflow (NEW — build itinerary) ← This amendment
Step 4E: Confirm Trip Readiness (NEW — gate before Deep Scan)
Step 4F: Run QuickScan OR DeepScan (was Step 4E)
________________


Step 4D: Draft Home Workflow
Entry Condition: User created trip in DRAFT state (Step 4C)
Goal: Build structured itinerary incrementally before coverage analysis
Sub-Steps:
4D.1 — Initial Capture (Voice or Manual)
User chooses:
                                                                                    * Option A: Voice narration ("Tell me about your trip")
                                                                                    * Option B: Manual entry ("Add leg")
Voice Path:
                                                                                    1. User taps mic
                                                                                    2. User narrates: "I'm flying from NYC to Paris on June 10..."
                                                                                    3. System parses → Proposes route segments
                                                                                    4. User confirms → Segments written to database
                                                                                    5. Trip sub-state: initial_capture → route_building
Manual Path:
                                                                                    1. User clicks "Add leg"
                                                                                    2. User fills form (Origin, Destination, Travel mode, Times)
                                                                                    3. User saves → Segment added
                                                                                    4. Trip sub-state: initial_capture → route_building
Outputs:
                                                                                    * Route segments created (route_segments table)
                                                                                    * Emit: route_segment_added
________________


4D.2 — Route Building
User adds segments until itinerary complete:
                                                                                    * Add more legs (manual or voice)
                                                                                    * Reorder segments (drag-drop)
                                                                                    * Edit segment times
System runs route validation (F-6.5.18):
                                                                                    * Check schedule conflicts
                                                                                    * Warn about tight connections
                                                                                    * Flag impossible travel times
User sees:
                                                                                    * Green ✓ (route valid)
                                                                                    * Yellow ⚠ (warnings, can acknowledge)
                                                                                    * Red ⛔ (errors, must fix)
Outputs:
                                                                                    * Route segments confirmed or edited
                                                                                    * Validation results displayed
                                                                                    * Emit: route_normalized
________________


4D.3 — Activity Planning (Optional)
System suggests activities (F-6.5.19):
                                                                                    * AI-inferred (based on destination + dates)
                                                                                    * Creator-sourced (from F-6.6.11, if user searched)
User reviews:
                                                                                    * Accept activity → Moves to itinerary
                                                                                    * Reject activity → Hidden from list
                                                                                    * Defer → Stays in suggestions
User may also:
                                                                                    * Add custom activities manually
                                                                                    * Skip activity planning entirely (proceed to readiness)
Outputs:
                                                                                    * Activity candidates created (activity_candidates table)
                                                                                    * Emit: activity_accepted, activity_rejected
Trip sub-state: route_building → activity_planning
________________


4D.4 — Resolve Blockers
System auto-flags blockers:
                                                                                    * Missing departure/return dates
                                                                                    * Route validation errors
                                                                                    * No route segments
User may add custom blockers:
                                                                                    * "Need to book hotel"
                                                                                    * "Check visa requirements"
User resolves:
                                                                                    * Click "Fix now" → Jumps to relevant screen (e.g., date picker, route editor)
                                                                                    * Mark done (user-added items)
Outputs:
                                                                                    * Unresolved items resolved (unresolved_items table)
                                                                                    * Emit: unresolved_item_resolved
Trip sub-state: activity_planning → readiness_review
________________


4D.5 — Check Readiness
User clicks: "Check Readiness"
System runs: evaluate_trip_readiness() (Section 3.5)
Result A (Not Ready):
                                                                                    * Show blockers list
                                                                                    * User must fix before proceeding
                                                                                    * Loop back to 4D.4
Result B (Ready):
                                                                                    * Show confirmation prompt: "Your trip is ready!"
                                                                                    * User clicks "Confirm trip is ready"
                                                                                    * Trip transitions: DRAFT → PRE_TRIP_STRUCTURED
                                                                                    * Emit: trip_readiness_confirmed
                                                                                    * Proceed to Step 4F (Deep Scan now unlocked)
Trip sub-state: readiness_review → confirmed
________________


User Paths Through Draft Home
Path 1: Voice-Only (Fastest)
1. Create trip (Step 4C)
2. Narrate entire itinerary (4D.1 Voice)
3. System parses, user confirms (4D.2)
4. Skip activities (4D.3)
5. Check readiness, confirm (4D.5)
6. Run Deep Scan (Step 4F)


Time: ~5 minutes
Path 2: Manual Entry (Most Control)
1. Create trip (Step 4C)
2. Add segments manually (4D.1 Manual, 4D.2)
3. Review AI activity suggestions, accept 3 (4D.3)
4. Fix route warning (tight connection) (4D.4)
5. Check readiness, confirm (4D.5)
6. Run Deep Scan (Step 4F)


Time: ~15 minutes
Path 3: Iterative (Resumable)
1. Create trip (Step 4C)
2. Add 2 segments, close app (4D.1-4D.2)
3. [Next day] Return, add 2 more segments (4D.2)
4. Add activities (4D.3)
5. Check readiness → Not ready (missing return date) (4D.5)
6. Add return date (4D.4)
7. Check readiness → Ready, confirm (4D.5)
8. Run Deep Scan (Step 4F)


Time: ~30 minutes (spread over days)
________________


Exit Conditions
Success Path:
                                                                                    * Trip in PRE_TRIP_STRUCTURED state
                                                                                    * Readiness confirmed
                                                                                    * Deep Scan unlocked
                                                                                    * Proceed to Step 4F
Abandon Path:
                                                                                    * User closes Draft Home without confirming
                                                                                    * Trip remains in DRAFT state
                                                                                    * Draft saved (all segments, activities, blockers persist)
                                                                                    * User can return later ("Continue planning")
________________


Cross-Flow Dependencies
Upstream:
                                                                                    * Step 4C (Create Trip) → Outputs trip in DRAFT state
Downstream:
                                                                                    * Step 4F (Deep Scan) → Requires trip in PRE_TRIP_STRUCTURED (readiness gate)
Parallel:
                                                                                    * Step 4B (Attach Policy) may happen before, during, or after Draft Home workflow
