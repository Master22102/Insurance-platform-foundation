## 1. Purpose

Capture revealed traveler preferences and organize them into three scopes:
what the traveler means **right now** (session), what applies to **this
trip** (trip), and what should be **remembered forever** (profile). The
onboarding SignalProfile is the seed; every subsequent interaction refines
it.

## 2. Data classification

- Profile preferences: **T2** (personal operational). RLS via
  `user_profiles` existing ownership policy.
- Carrier-exclusion preferences (`excluded_carriers`): T2, but especially
  sensitive ŌĆö never expose in any analytics or aggregate view.
- Any medical / mobility preference flagged with a T1 indicator: T1-tier
  treatment. Never logged, never exported without explicit user action.

## 3. Three memory layers

### 3.1 Session layer

Transient. React state or `sessionStorage`. Never written to the database.
Promoted to profile only if the user says yes to "Remember this for future
trips?".

### 3.2 Trip layer

`trips.metadata.preference_overrides` ŌĆö a partial preferences object scoped
to a single trip. Overrides profile values for that trip only. Expires when
the trip is archived.

```json
{
  "preference_overrides": {
    "accommodation": ["budget_ok_for_this_trip"],
    "transport": [],
    "food": [],
    "activities": [],
    "carriers_excluded": [],
    "timing": []
  }
}
```

### 3.3 Profile layer ŌĆö `user_profiles.preferences.preference_memory`

Array of preference-item objects. Each preference is explicit, sourced, and
individually active/inactive.

```json
{
  "preference_memory": [
    {
      "id": "e5d4b1fe-ŌĆ”",
      "category": "accommodation",
      "statement": "No hostels",
      "source": "voice_capture",
      "source_context": "trip_planning_session_42",
      "confirmed_at": "2026-04-21T00:00:00Z",
      "active": true
    }
  ]
}
```

Allowed `category` values: `accommodation`, `transport`, `food`,
`activities`, `carriers`, `timing`, `destinations`, `general`.
Allowed `source` values: `voice_capture`, `behavior_inferred`,
`incident_promotion`, `manual_entry`.
`active = false` is a soft delete.

## 5. How preferences are captured

### 5.1 Voice capture (FUTURE ŌĆö document, do not build yet)

Voice parse (across all 7 context types) detects patterns:
- "I don't like X" / "I hate X"
- "I prefer X" / "I love X"
- "I always want X" / "Never give me X"

On detection the system asks: "Remember for all trips, just this trip, or
just right now?" Routing:
- All trips ŌåÆ profile layer
- This trip ŌåÆ trip layer
- Right now ŌåÆ session layer (no persistence)

**Implementation note:** the first prompt builds preference storage, review
UI, and manual entry only.

### 5.2 Behavior inference (FUTURE)

After 3ŌĆō5 consistent choices (e.g., always rejecting hostels, always
picking afternoon flights), surface for confirmation:
"You've chosen afternoon flights on your last 5 trips. Should I remember
that you prefer avoiding early departures?"

Traveler options: Confirm ŌåÆ profile; Dismiss ŌåÆ never ask again; Not
always ŌåÆ keep observing.

### 5.3 Incident promotion (BUILD NOW)

F-6.5.8 captures incident-scoped preferences in
`incidents.preference_context` (`excluded_carriers`, `preferred_mode`,
`routing_preference`, `budget_signal`, etc.).

After the incident resolution flow completes, show a post-incident
promotion card:
- "During this incident, you mentioned these preferences:"
- Each preference with a checkbox
- "Remember for future trips" ŌåÆ POST each checked preference to profile layer
- "Skip ŌĆö these were just for this incident" ŌåÆ dismiss

Appears on the incident resolution success screen, after the claim packet
is generated. Requires a hook added to the incident resolution flow.

### 5.4 Manual entry (BUILD NOW)

User types preferences directly in the review page. Immediately stored.

## 6. Confirmation rules

**No preference is promoted to the profile layer without explicit user
confirmation.** The system may suggest, but it never silently writes.
Exception: onboarding SignalProfile writes are confirmed during onboarding.

## 7. RPCs (SECURITY DEFINER, search_path = '')

- `add_preference_to_memory(p_category text, p_statement text, p_source text,
  p_source_context text)` ŌĆö generates `id`, sets `confirmed_at`, appends to
  array, emits `preference_promoted_to_profile` or
  `preference_confirmed_by_user` depending on source.
- `update_preference_in_memory(p_id uuid, p_statement text, p_category text)` ŌĆö
  patches one array element, emits `preference_updated`.
- `remove_preference_from_memory(p_id uuid)` ŌĆö soft-delete (sets
  `active = false`), emits `preference_removed_by_user`.
- `list_preferences(p_category text default null)` ŌĆö returns active
  preferences, filtered when category supplied.
- `set_trip_preference_overrides(p_trip_id uuid, p_overrides jsonb)` ŌĆö
  validates schema, merges into `trips.metadata.preference_overrides`.

## 10. Deep Scan mode inference (FUTURE)

F-6.5.13 defines 6 scan modes. When the traveler initiates a Deep Scan,
Preference Memory checks accumulated preferences for mode signals via
keyword matching (not LLM inference):

- "love astronomy" + "off-the-beaten-path" + "hiking" ŌåÆ Adventure & Off-Path
- "budget hotels fine" + "cheap flights" ŌåÆ Cost Efficiency
- "no extreme sports" + "prefer quiet hotels" + "avoid crowds" ŌåÆ Rest & Recovery

The scan pre-selects the inferred mode; the traveler can accept or switch.
Deferred ŌĆö requires the Itinerary Optimizer (specced in F-6.5.13 but not
implemented).
