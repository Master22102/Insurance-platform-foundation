## 1. Purpose

Deep Scan is the platform's multi-axis trip-intelligence engine. It
synthesizes signals across the traveler's full itinerary — every leg,
every destination, every coverage layer — and returns structured,
confidence-labeled, clause-bound output covering vulnerabilities,
opportunities, regulatory requirements, and remediation guidance.

It is **not** prediction. It does not calculate probabilities. It
interprets signals and tells the traveler what to do with that knowledge.
Final determination rests with the relevant authority or provider.

## 2. Data classification

- Scan inputs (itinerary, attached policies): **T2**. Account-scoped.
- Scan outputs (opportunities, vulnerabilities, narratives): **T2**.
  Account-scoped. Export via Section 8.7 only.
- Per-region BD signal (thin-data gap log): **T3** platform analytics; no
  per-user data.

## 4. RLS (four policies per user-facing table)

Account-scoped SELECT on `deep_scan_runs`, `deep_scan_axis_outputs`,
`deep_scan_vulnerabilities`. INSERT/UPDATE via SECURITY DEFINER only.
No DELETE. Runs age out; outputs immutable.

## 5. Engine structure

- **Axis runners** in `lib/deep-scan/axes/axis-1.ts` through
  `axis-11.ts`. Each runner accepts the run context and returns a typed
  axis output. Axes 1–8 run unconditionally. Axes 9 and 10 run for
  international trips only. Axis 11 runs only when upstream signal
  detection fires.
- **Itinerary Optimizer** in `lib/deep-scan/optimizer.ts` weights outputs
  by the traveler's selected priority mode.
- **Narrative synthesizer (Axis 7)** consumes Axes 1–6 plus Coverage
  Graph + Causality Model output and emits per-leg plain-English
  briefings.
- **Vulnerability resolution** in `lib/deep-scan/resolution.ts` — at MVP
  runs unconditionally and emits `deep_scan_vulnerabilities` rows.
- **Confidence labels** attached to every interpretive output per
  Section 9.2.

## 6. RPCs (SECURITY DEFINER, search_path = '')

- `start_deep_scan(p_trip_id uuid, p_scan_type text, p_priority_mode text)` —
  validates credit balance, emits `deep_scan_initiated`, returns `run_id`.
- `consume_scan_credit(p_run_id uuid)` — **never** silent; require
  explicit user confirmation upstream. Emits `deep_scan_credit_consumed`.
- `record_axis_output(p_run_id uuid, p_axis int, p_confidence text, p_findings jsonb)` —
  idempotent upsert per `(run_id, axis)`.
- `finalize_deep_scan(p_run_id uuid)` — emits `deep_scan_completed`,
  triggers BD-signal extraction.

## 12. Acceptance criteria

- [ ] Quick Scan run consumes 0 credits and surfaces no confidence labels.
- [ ] Deep Scan run requires explicit credit confirmation; skipping it
      blocks the run without consuming credit.
- [ ] Axes 1–8 execute on every run; Axes 9–10 only on international
      trips; Axis 11 only when upstream signal detection fires.
- [ ] Every axis output carries a confidence label per Section 9.2.
- [ ] Boundary statement is present verbatim on every result view.
- [ ] Vulnerability resolution layer fires at MVP and attaches per-gap
      resolution steps.
- [ ] BD signal rows accumulate when any axis returns sparse data for a
      region.
- [ ] `deep_scan_credit_consumed` appears exactly once per consumption.
