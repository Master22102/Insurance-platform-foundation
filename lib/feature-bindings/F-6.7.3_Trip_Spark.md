## 1. Purpose

When a traveler's Field Notes cluster around a geographic region, Trip Spark
detects the concentration, estimates whether it is enough for a trip,
identifies what is missing, and offers to convert the cluster into a draft
trip — using the existing Trip Draft Engine, existing readiness checklist,
and optionally the traveler's budget preferences. It is **not** a parallel
trip model; it is a fast path into `ensure_trip_draft_version`.

## 2. Data classification

- Cluster records: **T2** (derived personal data). Account-scoped.
- Cluster coordinates / centroid: **T2** — never exposed via unauthenticated
  APIs.

## 4. RLS (four policies)

- SELECT: account membership.
- INSERT: SECURITY DEFINER only (clustering engine inserts; no direct
  `authenticated` policy).
- UPDATE: SECURITY DEFINER only for status transitions and score updates; a
  separate SECURITY DEFINER RPC permits user dismiss.
- DELETE: none. Clusters age out to `stale`; never deleted.

## 5. Clustering engine

### Trigger points
1. After `create_field_note` commits, enqueue `recompute_spark_clusters(account_id, country_code)`.
2. On Field Note update that changes `country_code` or `region`.
3. On Field Note deletion / archive.
4. Debounced on `/account/field-notes` page load (background, not on every
   render).

Clustering does NOT require user action to trigger.

### Algorithm (Phase 1 — deterministic, no ML)

1. Select all active (non-archived, non-converted) `field_notes` for the
   account, grouped by `country_code`.
2. Within each country, sub-group by region proximity:
   - If coordinates exist on notes: haversine distance with 100 km radius.
   - Otherwise: exact region-string match.
3. Compute per-cluster metrics:
   - `note_count` — member count
   - `category_count` — distinct categories across members
   - `geographic_coherence` — true if all members within 100 km OR share the
     same region string
   - `seasonality_overlap` — detect contradictory seasons
4. Trip-worthiness rule:
   `note_count >= 4 AND category_count >= 2 AND geographic_coherence = true`
5. Duration estimate bucket (displayed as a suggestion):
   - 4–6 notes → 3–4 days
   - 7–12 notes → 5–7 days
   - 13–20 notes → 8–10 days
   - 20+ notes → 10+ days
6. Missing-category analysis against the complete-trip template:
   `{ food, culture/activity, transit, lodging, scenic/nature }`.

Clusters that are trip-worthy but have seasonal conflicts get the split
prompt (Story 2 of the doctrine).

### Outputs
- `cluster_detected` event per new cluster.
- `cluster_trip_worthy` event the first time a cluster crosses the
  worthiness rule.
- `trip_spark_prompt_shown` when the banner renders on library / trips /
  dashboard.

## 6. RPCs (SECURITY DEFINER, search_path = '')

- `recompute_spark_clusters(p_account_id uuid, p_country_code text)` —
  idempotent reclustering; upserts clusters, updates counts / score /
  seasonal analysis.
- `convert_spark_cluster_to_trip(p_cluster_id uuid, p_selected_note_ids uuid[],
  p_duration_preference text, p_include_fillers boolean)` — calls
  `ensure_trip_draft_version` with destination_summary, PATCHes each selected
  Field Note with `attached_trip_id`, creates `activity_candidates` for each,
  fires checklist generation, kicks off Axis 5 pre-query. Sets
  `status = 'converted'`, `converted_trip_id`, `converted_at`.
- `dismiss_spark_cluster(p_cluster_id uuid, p_type text)` — `p_type IN
  ('not_now', 'permanent')`. `not_now` sets
  `dismissed_until = now() + interval '30 days'`. `permanent` sets
  `status = 'dismissed_permanent'` and blocks re-surfacing unless 2+ new
  notes join the cluster.

## 9. "Build this trip" flow (critical downstream)

Executed in order:

1. `ensure_trip_draft_version` with `destination_summary = "{region}, {country}"`
   and optional date window.
2. PATCH each selected Field Note with `attached_trip_id = new_trip_id`.
3. For each selected Field Note, create an `activity_candidate` row with
   `source='field_note'`, `activity_name=note.title`, `city=note.region`,
   `country_code=note.country_code`, `metadata.field_note_id=note.id`.
4. `generate_trip_checklist` for the trip's destinations → produces
   `participant_checklist_items` (visa, passport validity, health).
5. Deep Scan Axis 5 pre-query when F-6.5.13 active → surface 2–3 bonus finds
   as a toast after draft creation (`trip_spark_axis5_seed_fired`).
6. Navigate to `/trips/[new_trip_id]/draft`.

## 13. Stress behavior (§15.0)

Under PROTECTIVE: clustering computation pauses. Existing cluster prompts
remain visible but the Build action is disabled. No new trip drafts are
created automatically.

Stress scenario: 200+ Field Notes across 10 regions. Verify clustering
completes within 3 seconds, correct clusters detected, prompt appears for
the strongest cluster only.

## 14. Acceptance criteria

- [ ] Clustering is deterministic: same inputs yield identical cluster
      records on re-run.
- [ ] Trip-worthiness rule enforced (3 notes alone → no prompt; 4 notes
      single-category → no prompt).
- [ ] Threshold crossing fires `cluster_trip_worthy` exactly once per
      cluster until dismissed or converted.
- [ ] Convert action creates a trip draft via `ensure_trip_draft_version`
      with correct destinations and linked notes.
- [ ] Dismiss `not_now` re-surfaces after 30 days; `permanent` re-surfaces
      only if 2+ new notes added.
- [ ] Replay coverage: cluster lifecycle reconstructs from events.

## 15. Out of scope

- ML-based clustering across categories or across regions (Phase 2).
- Multi-account / shared cluster computation (never — account-scoped
  always).
- Automatic conversion without explicit user consent.
