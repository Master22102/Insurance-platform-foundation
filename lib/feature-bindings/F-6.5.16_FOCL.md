# F-6.5.16 — FOCL (Feature Operational Control Layer)

Canonical doctrine binding. Any future feature that lands on the platform MUST
pass through the checkpoints described here.

## Purpose

Founder-only operational cockpit. Surfaces state without duplicating business
rules. All mutations are recorded to the immutable Founder Action Ledger.

## Invariants (doctrine)

- I-01 Every registered feature is toggleable.
- I-02 Auto-rollback rule REQUIRED before rollout percentage > 0.
- I-03 Health probe required when rollout percentage > 0.
- I-04 Rollout scope is one of INCIDENT | USER | TRIP | REGION.
- I-05 Minimum mode is one of NORMAL | ELEVATED | PROTECTIVE | RECOVERY.
- I-06 Phase is one of MVP | Phase2 | Phase2Plus.
- I-07 Capability tier current ∈ [1, capability_tier_max].
- I-08 Connector status ∈ {not_required, required_unlicensed,
  licensed_not_integrated, active}.
- I-09 FOCL writes only via SECURITY DEFINER RPCs with `is_founder(auth.uid())`.
- I-10 All state changes append to `event_ledger` + `founder_action_ledger`.

- `feature_registered`, `feature_activation_state_changed`,

  `maintenance_window_opened`, `maintenance_window_closed`,
  `founder_action_recorded`.

## Registration Checklist (mandatory for new features)

Every new feature MUST satisfy the following before going live:

1. Row in `feature_registry` with: display_name, description, default_enabled,
   minimum_mode, phase, capability_tier_current, capability_tier_max,
   has_pending_extension, connector_status.
2. At least one row in `feature_rollout_rules` (auto-rollback condition).
3. Health probe metadata populated (`feature_rollout_health_state.metrics`).
4. Event types registered in `event_type_registry` with feature_id binding.
5. Binding file in `lib/feature-bindings/<feature_id>_<name>.md`.

Use `lib/focl/register-feature.ts::registerFeatureWithCheckpoints` or the SQL
RPC `public.register_feature_with_checkpoints(...)` to enforce steps 1–4
deterministically.

## Admin Gating

- UI: `/focl/*` routes reject non-founders at both (a) middleware (404-hide)
  and (b) `app/focl/layout.tsx` (client guard).
- DB: every FOCL write RPC runs `SECURITY DEFINER` and checks
  `public.is_founder(auth.uid())`. Direct-write RLS policies are NOT granted.

## Phone vs Laptop policy (doctrine)

- Phone: read-only EXCEPT urgent approvals (enforced in UI).
- Laptop: full control surface.
- Device hint: user-agent inspection at page level; does not relax server guards.
