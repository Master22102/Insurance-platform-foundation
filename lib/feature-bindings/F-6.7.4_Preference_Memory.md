# Feature Binding — F-6.7.4 Preference Memory

**Status:** Draft implementation prompt. Not yet built.
**Depends on:** `user_profiles.preferences` JSONB (exists), voice parse
pipeline (exists, 7 contexts), Budget Intelligence (F-6.7.3, parallel).
**Architectural note:** Extends the existing SignalProfile + voice

---

## 1. Purpose

Persistent, structured memory of traveler preferences that survives
across trips and is read by every planning surface. Today, voice-captured
preferences live in ephemeral artifacts tied to a single session. This
feature promotes them into a long-lived profile that the context engine,
checklist generator, Deep Scan, and Field Notes ranking consult.

## 2. Data classification

- Preference profile: **T2** (personal operational). RLS via
  `user_profiles` existing ownership policy.
- Any identity/health preference (dietary restriction with medical cause,
  mobility support needs): **T1-tier treatment when tagged**. Stored in
  `preferences.support_flags` with a flag indicating T1 handling; never
  logged, never exported without explicit user action.

## 3. Schema — JSONB contract under `user_profiles.preferences.memory`

```json
{
  "memory": {
    "version": 1,
    "updated_at": "2026-04-21T00:00:00Z",
    "travel_style": {
      "pace": "slow" | "balanced" | "packed",
      "planning_depth": "spontaneous" | "framework" | "detailed",
      "group_default": "solo" | "partner" | "family" | "friends",
      "repeat_destinations_ok": true
    },
    "interests": ["food", "nature", "history", "nightlife", "wellness"],
    "avoid_categories": ["extreme_adventure"],
    "lodging_style": {
      "prefers": ["boutique_hotel", "apartment_rental"],
      "avoids": ["hostel_dorm"]
    },
    "food": {
      "dietary": ["vegetarian"],
      "medical_restrictions": [],
      "adventurousness": "high"
    },
    "mobility": {
      "walking_tolerance_km_per_day": 12,
      "support_flags": []
    },
    "climate": {
      "comfort_min_c": 12,
      "comfort_max_c": 30,
      "humidity_tolerance": "medium"
    },
    "accessibility_needs": [],
    "sources": [
      {
        "path": "travel_style.pace",
        "captured_at": "2026-03-04T13:00:00Z",
        "context": "onboarding_round_2",
        "source_type": "voice_parse"
      }
    ]
  }
}
```

Every leaf field that is written carries a provenance entry in `sources`.
This is how Preference Memory stays auditable — any UI reading from it can
surface "we learned this from your onboarding voice session".

A JSON schema at `lib/schemas/preference-memory.schema.json` is validated
before writes.

## 4. RPCs

- `set_preference_memory(p_patch jsonb, p_source_context text)` — accepts
  a partial patch (RFC 7396 JSON Merge Patch semantics), validates merged
  result against schema, writes to `preferences.memory`, appends
  provenance entries for each changed leaf.
- `get_preference_memory()` — returns current structure; fills defaults
  for missing keys.
- `clear_preference_memory_field(p_json_path text)` — removes a leaf and
  logs an event; used for "forget this" UX.

All SECURITY DEFINER with search_path = ''.

## 5. Migration

`<ts>_f_6_7_4_preference_memory.sql`:
- Adds a CHECK: if `preferences -> 'memory' -> 'version'` present, integer
  >= 1.
- Adds expression indexes on commonly-joined leaves:
  - `((preferences -> 'memory' -> 'travel_style' ->> 'pace'))`
  - `((preferences -> 'memory' -> 'climate' ->> 'comfort_min_c'))`
- No new tables.

## 6. Event types (feature_id F-6.7.4)

- `preference_memory_set` (info)
- `preference_memory_updated_from_voice` (info)
- `preference_memory_field_cleared` (info)
- `preference_memory_conflict_detected` (warn) — emitted when an incoming
  patch would overwrite a field with a contradictory value within 24h of
  the previous write; requires UI confirmation

## 7. UI surfaces

- `/account/preferences/memory` — structured editor grouped by
  sub-profile (style, interests, lodging, food, mobility, climate).
- Voice capture (`preference_capture` context) — matched utterances
  produce `set_preference_memory` patches; UI confirms before persisting
  any field flagged T1 (medical/mobility).
- Context engine (`lib/context-engine/evaluate.ts`) — reads memory on
  every evaluation; can weight recommendations by pace, interests,
  climate tolerance.
- Deep Scan connectors — Axis 5 and others receive memory as input to
  rank candidates (e.g. prefer bioluminescence over extreme sports for a
  traveler with `avoid_categories: ["extreme_adventure"]`).
- Checklist generation — mobility and dietary flags generate specific
  items (e.g. step-free hotel confirmation, dietary card translation).

## 9. Cross-feature dependencies

Writes: `user_profiles.preferences.memory`, `event_ledger`.
Reads: `user_profiles`, `voice_artifacts`.
Consumed by:
- **Context Engine (F-6.6.14)** — weights all 8 context states.
- **Deep Scan (F-6.5.13)** — axis ranking inputs.
- **Checklist Screen Family** — mobility / dietary / climate items.
- **Budget Intelligence (F-6.7.3)** — `travel_style.pace` informs per-day
  activity / food base assumptions when `budget` itself is unset.
- **Trip Spark (F-6.7.2)** — `interests` and `avoid_categories` weight
  cluster scoring.

## 10. Acceptance criteria

- [ ] `set_preference_memory` with a patch that violates the schema
      returns an error and does NOT write.
- [ ] Every leaf write appends exactly one provenance entry.
- [ ] Voice capture writes are idempotent on replay — identical payload
      with identical captured_at produces no duplicate entry.
- [ ] Conflict detection fires `preference_memory_conflict_detected`
      exactly once and blocks overwrite until UI confirmation.
- [ ] Clearing a field removes it from the JSON and emits
      `preference_memory_field_cleared` with the cleared path.
- [ ] Replay coverage: memory state reconstructs from events without
      requiring the current JSON snapshot.

## 11. Out of scope

- Cross-account profile sharing (never — account-scoped).
- Automatic preference inference from trip history (Phase 2; requires
  historical analysis and an ADR).
- AI-generated preference summaries for traveler self-review (Phase 2).
