# Feature Binding — F-6.7.4 Budget Intelligence (Category-Level Spend Model)

**Status:** Draft implementation prompt. Not yet built.
**Source doctrine:** `lib/Doctrine/Feature ID: F-6.7.4 — Budget Intelligence` (v2.0, April 2026).
**Depends on:** `user_profiles.preferences` JSONB (exists), `trips.metadata`
JSONB (exists), context engine `lib/context-engine/evaluate.ts` (exists).
**Architectural rule:** **No new table.** Storage lives inside existing JSONB
fields. Any prior draft that invented a `destination_cost_cache` or similar
reference table is retired.

> Numbering history: an earlier draft bound Budget Intelligence as F-6.7.3.
> Canonical doctrine places it at **F-6.7.4**. See MISMATCH_LOG M-007.

---

## 1. Purpose

Store the numeric, conditional, per-trip-overridable shape of a traveler's
budget so that every downstream surface — Trip Spark, context engine nudges,
Deep Scan financial commentary, the eventual Stay Range Engine — can make
specific recommendations rather than asking for a bucket label.

## 2. Data classification

- Budget preferences: **T2** (personal operational data). Financial
  preference data is personal; numbers reveal spending capacity.
- Field-level treatment: `lodging_per_night_cap` and `flight_pain_threshold`
  are the most revealing values. Do not expose in logs, events, or error
  messages.
- **No historical actual-spend tracking** in MVP. T1 handling blocked on
  ADR.

## 3. Storage — JSONB contracts

### 3.1 Profile layer — `user_profiles.preferences.budget_intelligence`

```json
{
  "budget_intelligence": {
    "lodging_per_night_cap": 180,
    "lodging_splurge_ceiling": 250,
    "lodging_splurge_frequency": 2,
    "food_daily_comfort": 65,
    "activity_daily_comfort": 40,
    "flight_pain_threshold": 600,
    "train_vs_flight_bias": "prefer_train",
    "total_daily_target": 250,
    "travel_mode": "couple",
    "domestic_vs_international": "international",
    "currency": "USD",
    "last_updated": "2026-04-21T00:00:00Z"
  }
}
```

All fields are nullable. Field semantics and allowed values are defined in
the source doctrine §3.2.

### 3.2 Trip-layer override — `trips.metadata.budget_override`

Partial `BudgetIntelligence` object. Trip-level values win over profile-level
values when present.

```json
{
  "budget_override": {
    "lodging_per_night_cap": 220
  }
}
```

### 3.3 Migration

Single migration `<ts>_f_6_7_4_budget_intelligence.sql`:
- CHECK constraint: when `preferences -> 'budget_intelligence' -> 'currency'`
  is present, it is a 3-char ISO 4217 code.
- Expression index on
  `((preferences -> 'budget_intelligence' ->> 'travel_mode'))` for analytics.
- No new table.

JSON schemas live at `lib/schemas/budget-intelligence.schema.json` and
`lib/schemas/budget-override.schema.json` and are validated by the RPCs
below before writing.

## 4. RPCs (SECURITY DEFINER, search_path = '')

- `set_budget_preferences(p_budget jsonb)` — validates against schema,
  merges into `user_profiles.preferences.budget_intelligence`, sets
  `last_updated` server-side, emits `budget_preferences_updated`.
- `set_trip_budget_override(p_trip_id uuid, p_override jsonb)` — validates
  partial schema, merges into `trips.metadata.budget_override`, emits
  `budget_override_set`.
- `get_budget_for_trip(p_trip_id uuid)` — returns the effective budget
  (profile merged with override) for the trip, used by Trip Spark and the
  context engine.

## 5. Context engine integration

New branch in `lib/context-engine/evaluate.ts`. Fires within the existing
`pre_trip` context state — **not** a new state.

Fire conditions:
- `maturity_state` is `DRAFT` or `PRE_TRIP_STRUCTURED`.
- `user_profiles.preferences.budget_intelligence` exists.
- Trip has at least one destination with known regional cost data.

Outputs (examples):
- "Your lodging target may be tight for {destination} during {season}.
  Consider nearby alternatives or adjusting your budget for this trip."
- "{Destination} tends to run above your daily target. The platform will
  help you find options that fit."

Failure mode rule: if budget JSONB is null or malformed, the branch is a
no-op. `console.warn` and skip. **Never crash the context engine.**

## 6. Event types (feature_id F-6.7.4)

- `budget_preferences_updated` (info) — `{ account_id, fields_changed }`
- `budget_override_set` (info) — `{ trip_id, account_id, fields_overridden }`
- `budget_signal_captured` (info) — future, when voice capture is built

## 7. UI surfaces

- `/account/preferences/budget` (Surface ID `S-BUDGET-001`):
  - Lodging section: cap, splurge ceiling, splurge frequency stepper
  - Food and Activities sections: daily comfort amounts
  - Transport section: flight pain threshold + train/flight bias radios
  - Overall section: total daily target
  - Travel style: solo / couple / family / group
  - Domestic / international / both radios
  - Currency dropdown (USD, EUR, GBP, JPY, AUD, CAD, CHF, SGD)
  - All fields optional. Loads current values, empty when unset.
- `app/(app)/account/page.tsx` — new "Budget preferences" row with summary
  line (`"$180/night · $65/day food · couple"`) or `"Not set"` if unset.
- Trip workspace per-trip override card — **documented as planned,
  deferred** in v1.

## 9. Voice capture (PLANNED — document, do not build yet)

The voice parse route should detect budget statements across all context
types (not just `signal_capture`). Patterns: "I don't want to spend more
than $X", "My budget is $X a day", "I'd rather not go over $X for flights."

When detected, the system asks "Should I remember this for future trips, or
just this one?" and routes to profile or trip-level storage. Implementation
deferred.

## 13. Acceptance criteria

- [ ] `set_budget_preferences` rejects payloads that violate the schema.
- [ ] Empty/unset budget renders the preferences page without error.
- [ ] Setting lodging cap = 200, food = 65, save → reload persists values.
- [ ] Clearing all fields makes the context branch a no-op.
- [ ] `get_budget_for_trip` returns profile values when no override exists
      and overridden values when override is present (trip wins).
- [ ] RLS enforcement: user B cannot read user A's budget via API.
