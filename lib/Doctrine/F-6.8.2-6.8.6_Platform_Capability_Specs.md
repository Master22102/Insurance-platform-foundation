# F-6.8.2 — Multi-Modal Transit Visualization

## 1. Why This Exists

The platform's data model supports five travel modes (`air`, `rail`, `sea`, `road`, `mixed`) via `trips.travel_mode_primary` and segment-level mode fields. The initial Visual Itinerary Timeline (F-6.8.1) only visually differentiated air transit. Every other mode rendered identically. This breaks the promise of the platform supporting road trips, rail tours, cruise itineraries, and mixed-mode journeys.

Christian's own itinerary is the proof case: it includes air (JFK → BKK → DPS → SIN), rail (Ayutthaya day trip, MRT day passes, Eurail in Europe trip), sea (longtail boats, fast boat to Gili), and road (private car, scooter rental, airport transfers). Every mode needs its own visual language.

## 2. What to Build

### 2.1 Mode Tagging on Transit Cards

Every card with `type = 'transit'` must have a `mode` field: `air | rail | sea | road | mixed`. Add to `activity_candidates` table:

```sql
ALTER TABLE public.activity_candidates
  ADD COLUMN IF NOT EXISTS transit_mode text
  CHECK (transit_mode IN ('air', 'rail', 'sea', 'road', 'mixed'));
```

For existing `route_segments` table, verify the `mode` column exists (it does based on repo inspection); otherwise mirror the same constraint.

### 2.2 Visual Treatment Per Mode

Each mode gets a distinct color and icon in the card rendering:

| Mode | Icon | Color | Hex |
|---|---|---|---|
| Air | ✈ | Blue | `#60a5fa` |
| Rail | 🚆 | Green | `#34d399` |
| Sea | ⛴ | Cyan | `#22d3ee` |
| Road | 🚗 | Orange | `#fb923c` |
| Mixed | 🔀 | Purple | `#a78bfa` |

A `<span class="transit-mode-badge transit-mode-{mode}">` appears next to the time slot on transit cards. Border-left color on the card remains type-based (blue for transit), but the mode badge adds secondary signal.

### 2.3 Transit Break Indicators (Between Destinations)

The transit break that appears between destination day groups (e.g., Bangkok → Bali) must render per-mode:
- Air: blue dashed border, ✈ icon, "Thai Airways flight · 4h 30m"
- Rail: green dashed border, 🚆 icon, "Eurail · 6h 20m · reserved seat"
- Sea: cyan dashed border, ⛴ icon, "Catamaran ferry · 2h 15m"
- Road: orange dashed border, 🚗 icon, "Rental car · 4h 45m drive · 312 miles"

### 2.4 Mode Legend in Hero

Next to "17 items planned" in the hero metadata row, show a mini mode legend — small icons for every mode used in the trip followed by "mixed modes" text. If the trip is single-mode (all air), hide the legend.

### 2.5 Per-Mode Readiness Implications

Each mode triggers different readiness checks:
- **Air:** passport validity, visa requirements, medication import rules, baggage restrictions
- **Rail:** seat reservations, station changes (Eurostar check-in varies), onboard meal bookings
- **Sea:** embarkation documents, cabin class, port schedules, seasickness meds, cruise contract extraction
- **Road:** driver's license validity, IDP (International Driving Permit), insurance coverage for rental, one-way vs round-trip return fees

The readiness checklist engine (F-6.5.9) must branch on transit mode when generating items.

## 3. §9a Declaration

- Trip workspace: **YES** — visual changes to timeline cards
- Readiness: **YES** — different modes trigger different checklist items
- Right Now / context: **YES** — context messaging varies by mode (road = "check fuel + rest stops"; sea = "confirm embarkation time")
- FOCL / activation: **NO** — view layer only

## 4. Downstream Impact

Cruise (sea mode) specifically needs: shore excursion coverage logic, missed port detection as disruption type, medical evacuation at sea specifics, passenger cruise contracts. These are already documented gaps — multi-modal visualization exposes them visually.

Road mode specifically needs: route optimizer integration (Wanderlog-style), fuel cost estimation, toll calculation, rest stop suggestions for long drives. All deferred to a future Road Trip Mode spec.

# F-6.8.3 — Offline Mode

## 1. Why This Exists

Travelers lose connectivity. International data roaming is expensive or unreliable. Wi-Fi in remote areas is spotty. The platform must be usable when offline for the critical actions: viewing itinerary, showing documents, capturing incidents, accessing safety card, displaying booking confirmations.

The repo has `navigator.onLine` checks in TravelShield and notifications but no comprehensive offline strategy, no service worker, no IndexedDB cache. This is a production-blocker for international travel use.

## 2. What to Build

### 2.1 Service Worker Registration

Register a service worker at `/sw.js` that caches:
- All static assets (CSS, JS, fonts, app shell)
- Trip data for trips marked as "active" (departure within 7 days OR actively traveling)
- All personal documents and safety vault content
- Policy PDFs and extracted coverage data
- Destination intelligence already fetched for active trip countries

### 2.2 Cache Strategy

**Cache-first with network fallback** for:
- Trip pages (`/trips/*`)
- Document viewer (`/account/documents/*`)
- Safety card (`/trips/*/safety`)
- Coverage summary (`/trips/*/coverage`)

**Network-first with cache fallback** for:
- Real-time features (TravelShield check-ins, live pricing, current weather)
- Context engine Right Now panel
- Deep Scan results (time-sensitive)

### 2.3 Offline Indicators

When `navigator.onLine === false`:
- Global banner at top of app: "You're offline. Showing cached trip data from [timestamp]."
- Features requiring network show a "sync when online" badge
- Voice narration: queue recordings locally, upload when connection returns
- Incident capture: store locally in IndexedDB, sync to Supabase when online
- Budget expense logging: queue locally, sync on reconnect

### 2.4 Explicit Offline Downloads

On the trip detail page, add a "Download for offline" button:
- Downloads: all day photos, all documents, coverage summary, safety card, destination intelligence, emergency contacts
- Estimated storage: ~15-30 MB per trip
- Shows progress indicator during download
- Downloaded trips get an offline-ready badge

### 2.5 IndexedDB Schema

```javascript
// wayfarer_offline database
{
  trips: { trip_id, trip_data, cached_at, photos_cached: [], documents_cached: [] },
  segments: { segment_id, trip_id, segment_data },
  activity_candidates: { candidate_id, trip_id, data },
  documents: { document_id, trip_id, blob, mime_type },
  pending_syncs: { sync_id, entity_type, entity_data, created_at, retries }
}
```

### 2.6 Sync Queue

Actions taken offline go into `pending_syncs`. When online:
1. Iterate queue in FIFO order
2. POST each action to the appropriate API endpoint
3. On success, remove from queue
4. On conflict (e.g., trip modified by another collaborator), surface merge dialog
5. On network error, retry with exponential backoff, max 5 attempts

### 2.7 What NOT to Offline

- Payment flows (always require network)
- Policy purchasing (always require network)
- Shared group editing (requires real-time sync)
- Claim packet submission (requires server-side PDF generation)

## 3. §9a Declaration

- Trip workspace: **YES** — trips must render from cached data
- Readiness: **YES** — readiness checks cached with trip
- Right Now / context: **YES** — degraded experience offline, shows "Last updated" timestamps
- FOCL / activation: **YES** — new capability requires FOCL gate `offline_mode_enabled`

## 4. Stress Behavior

Under PROTECTIVE mode, offline features remain available (they don't hit the server). Under surge, prioritize offline-first renders to reduce server load.

# F-6.8.4 — Map View

## 1. Why This Exists

Users expect to see their trip on a map. Wanderlog's map integration is table stakes for the category. The current "View on map" button in Wayfarer is a toast — no actual map screen.

Two use cases, two different implementations:
1. **Planning view** (in-app map): see your route and pins while planning, styled consistently with Wayfarer's navy/gold aesthetic
2. **Navigation handoff** (external): tap "Open in Google Maps" to navigate to a specific location using the user's preferred map app

## 2. What to Build

### 2.1 Map Library Choice

**Recommendation:** Mapbox GL JS for in-app planning view. Rationale:
- Custom styling (navy theme with gold routes matches the design language)
- Interactive vector tiles perform well on desktop
- React wrapper `react-map-gl` is well-maintained
- Free tier: 50,000 loads/month for planning use
- Alternative: Leaflet + OpenStreetMap (free, less polished, heavier styling work)

### 2.2 Planning View Screen

Route: `/trips/[trip_id]/map` (new surface: S-MAP-001)

Layout:
- Full-screen map (Mapbox GL)
- Left overlay panel (collapsible): list of all destinations, activities, accommodations as pins
- Right overlay panel: selected pin details card
- Top bar: "← Back to timeline" · day filter chips · search
- Bottom bar: "Open in Google Maps" (launches external app)

Pin styling:
- Destination pins: gold circles with country flag, numbered (1, 2, 3 for visit order)
- Activity pins: orange droplets with activity icon
- Accommodation pins: purple droplets with bed icon
- Dining pins: red droplets with fork icon

Route lines between destinations:
- Gold solid line for air segments (showing great circle path)
- Green solid line for rail
- Cyan dashed line for sea
- Orange solid line for road (actual driving route via Mapbox directions API)

### 2.3 External Map Handoff

Every location-bearing card (accommodation, activity, dining) gets an "Open in Maps" button:
- iOS: `maps://?q={address}` opens Apple Maps
- Android: `geo:?q={address}` opens default map app
- Web: `https://www.google.com/maps/search/?api=1&query={encoded_address}`

Detection via `navigator.userAgent` or feature detection. Deep link preserves the query string.

### 2.4 Day Filter

Chips at the top of the map filter pins by day:
- "All" (default)
- "Day 1", "Day 2", ... (one per day)
- Selected day: pins highlighted, others dimmed
- Route line only shows between consecutive visited locations for that day

### 2.5 Search

Search bar on the map (top-right): searches:
- Current trip's pins first (fast local filter)
- Then Mapbox Geocoding for any world location
- Result list shows: name, distance from current center, tap to zoom

### 2.6 Offline Behavior

When offline:
- Map tiles show cached zones (last 7 days of panning cached automatically by Mapbox GL)
- Pins render from cached trip data
- Directions API unavailable — show static great-circle lines
- "Open in Maps" still works if the external map app has offline support

### 2.7 Schema

```sql
-- Add to activity_candidates and route_segments
ALTER TABLE public.activity_candidates
  ADD COLUMN IF NOT EXISTS latitude numeric,
  ADD COLUMN IF NOT EXISTS longitude numeric,
  ADD COLUMN IF NOT EXISTS address text;

ALTER TABLE public.route_segments
  ADD COLUMN IF NOT EXISTS origin_lat numeric,
  ADD COLUMN IF NOT EXISTS origin_lng numeric,
  ADD COLUMN IF NOT EXISTS destination_lat numeric,
  ADD COLUMN IF NOT EXISTS destination_lng numeric;
```

Geocoding happens at import time (Tier 2/3 LLM can extract addresses) or via an on-demand geocoding pass when the user opens the map view for the first time.

### 2.8 Cost Model

Mapbox GL free tier: 50,000 loads/month. At scale: $0.50 per 1,000 loads above that. Expected usage: ~2-3 map loads per active trip. For 1,000 users with 1.5 trips/year, ~4,500 loads/year. Well within free tier.

## 3. §9a Declaration

- Trip workspace: **YES** — new surface, linked from timeline
- Readiness: **NO**
- Right Now / context: **NO** — separate surface
- FOCL / activation: **YES** — `map_view_enabled` feature gate

# F-6.8.5 — Expense Tracking During Trip

## 1. Why This Exists

The Budget Intelligence spec (F-6.7.4) handles pre-trip budget planning — setting caps, per-destination overrides, category allocations. It does NOT handle during-trip expense logging. When you spend $65 on dinner in Bangkok, nothing captures that.

Wanderlog has expense tracking with split-bill features. Without this, our budget panel shows plan-vs-plan, not plan-vs-actual. The Coverage Intelligence Loop also needs actual spending data to trigger claim opportunities (e.g., "You spent $340 on a hotel during the delay — claim the trip delay coverage").

## 2. What to Build

### 2.1 Schema

```sql
CREATE TABLE IF NOT EXISTS public.expense_log (
  expense_id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  trip_id uuid NOT NULL REFERENCES public.trips(trip_id) ON DELETE CASCADE,
  account_id uuid NOT NULL,
  category text NOT NULL CHECK (category IN (
    'flights', 'accommodation', 'activities', 'dining', 'transport', 'shopping', 'insurance', 'other'
  )),
  amount numeric NOT NULL,
  currency text NOT NULL DEFAULT 'USD',
  amount_usd numeric,
  description text,
  linked_card_id uuid REFERENCES public.activity_candidates(item_id),
  receipt_image_url text,
  expense_date date NOT NULL,
  location_city text,
  location_country text,
  created_at timestamptz NOT NULL DEFAULT now(),
  source text CHECK (source IN ('manual', 'voice', 'receipt_scan', 'card_sync'))
);

ALTER TABLE public.expense_log ENABLE ROW LEVEL SECURITY;
CREATE POLICY expense_log_owner ON public.expense_log
  FOR ALL TO authenticated USING (account_id = auth.uid());
```

### 2.2 Capture Flows

**Manual entry:** "+ Add expense" button on trip detail page → form with category dropdown, amount, currency, description, date, optional photo attachment.

**Voice capture:** Mic button in expense modal — user says "I just spent 85 dollars on dinner at the market" → voice parse extracts amount ($85), category (dining), description (market dinner) → pre-fills form.

**Receipt scan:** User takes a photo of a receipt → OCR pipeline extracts total, date, merchant → pre-fills form. Uses existing document-intelligence infrastructure. Implementation: reuse the PDF text extraction pattern, adapted for receipt images (Tesseract.js client-side for small receipts, server-side for large ones).

**Linked-to-card:** If the expense is tied to a planned card (e.g., the Taj Mahal tour you budgeted $99 for actually cost $112), link via `linked_card_id`. The card's display shows both: planned $99 · actual $112.

### 2.3 Currency Conversion

Use a free FX API (e.g., ExchangeRate-API free tier: 1,500 requests/month) to convert to USD at transaction time. Cache daily rates in a `fx_rates` table to reduce API calls.

### 2.4 Budget Panel Updates

The Budget Breakdown panel (already designed in F-6.8.1) shows spent-vs-allocated per category:
- Green bar: under budget (e.g., 40% of accommodation spent, 55% of trip elapsed)
- Amber: on track (spending roughly matches trip progress)
- Red: over budget
- The progress bar color changes based on spend rate vs. trip timeline

### 2.5 Per-Destination Spending

Within a destination, tally actual expenses. Compare to destination_cost_cache estimates. Flag when a destination is significantly over typical spending — possible trigger for Deep Scan hidden_opportunity axis ("Hotels in Bangkok averaging $180/night above typical — consider rebooking last 2 nights at a different property").

### 2.6 Coverage Claim Opportunity Detection

When actual spending on accommodation during a delay window exceeds what insurance would cover:
- Pattern: `expense_log.expense_date` between `incidents.disruption_started_at` and `incidents.resolved_at` AND category matches coverage trigger
- Surface a card: "You spent $340 on hotels during the 18-hour delay. Your policy covers up to $500 for delays over 6 hours. Want to file a claim?"
- Links to existing claim packet generator with expense data pre-loaded

### 2.7 Split with Companions

For group trips (F-6.6.x), expenses can be marked "split evenly" or "I paid for group":
- Adds `expense_shares` table: `(expense_id, account_id, share_amount, paid_by_account_id)`
- End-of-trip settlement view shows who owes whom
- Simple splitwise-like math, no payment processing

### 2.8 API Routes

- `POST /api/trips/[trip_id]/expenses` — create expense
- `GET /api/trips/[trip_id]/expenses` — list, filter by category/date
- `PATCH /api/trips/[trip_id]/expenses/[expense_id]` — update
- `DELETE /api/trips/[trip_id]/expenses/[expense_id]` — delete
- `GET /api/trips/[trip_id]/expenses/summary` — aggregated totals by category, by destination

## 3. §9a Declaration

- Trip workspace: **YES** — new capture UI, budget panel updates
- Readiness: **NO**
- Right Now / context: **YES** — "you might want to log that expense" nudges during trip
- FOCL / activation: **YES** — `expense_tracking_enabled`

## 4. Offline Behavior (Cross-Cut with F-6.8.3)

Expense entries work fully offline. Stored in IndexedDB `pending_syncs`, uploaded when connection returns. Photo receipts stored as blobs until upload.

# F-6.8.6 — Gmail / Email Itinerary Import

## 1. Why This Exists

Wanderlog's killer onboarding feature is Gmail integration: users connect their inbox and flight/hotel confirmations auto-populate their trip. Our current import is PDF/DOCX/TXT/ICS/narration — all manual effort.

Most travelers never print their confirmations; they live in email. Reducing import friction from "upload document" to "forward email" or "connect inbox once" dramatically increases conversion.

## 2. What to Build

### 2.1 Two Paths, One Outcome

**Path A: Forward-to-import (simple, launch-ready)**
- User forwards any confirmation email to `import@wayfarer.app`
- Inbound email webhook (via SendGrid, Postmark, or AWS SES) routes to `/api/inbound/email`
- Match sender email to user's account via `account_emails` table
- Parse email using Tier 2/3 LLM pipeline (same as PDF import)
- Create or update trip draft
- Notify user: "We imported your flight from that United confirmation. Review on your trip."

**Path B: Gmail OAuth connection (powerful, deeper integration)**
- User grants OAuth permission to read their Gmail
- Scope: `gmail.readonly`
- Scan inbox for confirmation emails (travel labels, known sender domains)
- Auto-import confirmations as they arrive via Gmail push notifications (webhook)
- Store last-scanned timestamp, never re-scan the same message

Launch with Path A. Add Path B in a later phase (OAuth requires Google verification which takes weeks).

### 2.2 Known Sender Patterns

Maintain a curated list of travel-related sender domains:
- Airlines: `delta.com`, `united.com`, `aa.com`, `emirates.com`, etc.
- Hotels: `marriott.com`, `ihg.com`, `hilton.com`, `booking.com`, `airbnb.com`, etc.
- Car rentals: `hertz.com`, `enterprise.com`, `turo.com`, etc.
- Tour operators: `viator.com`, `getyourguide.com`, etc.
- Cruise: `royalcaribbean.com`, `carnival.com`, etc.

Table: `sender_patterns` with columns `(pattern, category, priority)` — pattern can be domain or specific address.

### 2.3 Email Parsing Pipeline

Leverages the existing itinerary parser tier system:
1. **Tier 1 (regex):** Parses subject line for known patterns ("Flight confirmation", "Reservation confirmed at [hotel]"). Extracts basic fields.
2. **Tier 2 (Sonnet):** Full email body parsing for typical confirmations (single flight, single hotel booking, etc.). ~$0.02 per email.
3. **Tier 3 (Opus):** Complex multi-segment itineraries from travel agents, multi-hotel package bookings. Rare.

Complexity scoring: most single-confirmation emails stay on Tier 2.

### 2.4 Matching to Existing Trips

Email arrives → extracted dates/destinations checked against existing draft trips:
- If dates and primary destination overlap → append to that trip as a new segment
- If no match → create new trip draft, notify user
- If ambiguous (two overlapping trips) → surface disambiguation UI

### 2.5 Schema

```sql
CREATE TABLE IF NOT EXISTS public.account_emails (
  account_email_id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id uuid NOT NULL,
  email_address text NOT NULL UNIQUE,
  verified_at timestamptz,
  allow_import boolean NOT NULL DEFAULT true,
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE IF NOT EXISTS public.email_imports (
  import_id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  account_id uuid NOT NULL,
  source text NOT NULL CHECK (source IN ('forward', 'gmail_oauth')),
  from_address text,
  subject text,
  received_at timestamptz NOT NULL,
  parsed_trip_id uuid REFERENCES public.trips(trip_id),
  parse_tier integer,
  parse_cost_usd numeric,
  parse_status text CHECK (parse_status IN ('pending', 'success', 'partial', 'failed')),
  raw_email_storage_key text,
  created_at timestamptz NOT NULL DEFAULT now()
);
```

### 2.6 Privacy & Security

- OAuth tokens encrypted at rest
- Users can revoke Gmail access at any time
- Forwarded emails retained 30 days then purged
- Never scan personal emails (only travel-related senders)
- Audit log of every email processed
- Clear UI showing what was imported from which email

### 2.7 User-Facing Copy

Import source page:
- "Forward a confirmation → `import@wayfarer.app`" (quickest)
- "Upload a PDF / document" (current)
- "Connect your Gmail" (coming soon)
- "Tell us about your trip" (narration)
- "Enter manually" (last resort)

### 2.8 Inbound Webhook Vendor Choice

Recommend **Postmark** for inbound email:
- $1.25 per 1,000 inbound emails
- Easy webhook setup (`import@wayfarer.app` routes to your endpoint)
- Strong deliverability, good documentation
- Alternatives: SendGrid Inbound Parse (free up to 100 emails/day), AWS SES ($0.10 per 1,000)

### 2.9 Cost Model

Per 1,000 users with average 3 imports each per year = 3,000 email imports/year.
- Postmark inbound: $3.75/year
- Tier 2 Sonnet parsing: $60/year ($0.02 × 3000)
- Total: ~$64/year for email imports at 1,000-user scale. Trivial.

## 3. §9a Declaration

- Trip workspace: **YES** — imports create/update trips
- Readiness: **YES** — imported reservations trigger readiness checks
- Right Now / context: **NO**
- FOCL / activation: **YES** — `email_import_enabled` feature gate. Gmail OAuth gated separately (`gmail_oauth_enabled`).
