Feature ID: F-6.5.7
Incident Query & Retrospective Engine
Version 1.0 | March 2026 | Phase 1 (MVP) Feature
User Problem Solved
Founder Problem: "I should be able to say, 'Hey, didn't something happen last Thursday that put us in red alert?' And the system should figure that out easily and tell me what happened, why it happened, was something done automatically to patch it, or does something still need to be done."
Platform incidents generate hundreds of event ledger entries. Without natural language query and auto-narrative generation, the Founder must:
* Manually search event logs (technical, time-consuming)
* Reconstruct incident timelines from scattered events
* Determine which incidents require follow-up vs which auto-recovered
* Remember patterns across weeks/months ("Didn't this happen before?")
In one sentence: "Ask the platform 'What happened last Thursday?' in plain English, and get a founder-friendly narrative explaining what went wrong, what got fixed automatically, what still needs attention, and whether this is part of a pattern."
________________


Purpose
Incident Query & Retrospective Engine serves four goals:
1. Natural Language Incident Search: Query incidents conversationally ("What happened last Thursday?", "Show me all red alerts from March", "Why did we enter Safe Mode?")
2. Auto-Generated Narratives: Convert technical event ledger data into founder-friendly stories with context, causality, and outcomes
3. Platform Memory / Pattern Detection: Identify recurring issues across multiple time windows (30 days, 90 days, 365 days) and suggest mitigations
4. Audit Trail & Queryability: Every incident fully documented, easily searchable, exportable for compliance/retrospectives
NOT an Analytics Dashboard: This is incident-specific storytelling, not broad platform metrics (those go in weekly digest, FOCL dashboards).

Failure Modes & Safe Defaults
1. Query Returns Too Many Results
Failure: "Show me all incidents from March" returns 500 events.
Safe Default:
* Aggregate by PRIMARY incident (e.g., "3 Safe Mode episodes, 47 Orange alerts")
* Surface top 5 incidents by severity/cost
* Offer filter: "Showing 5 most significant. [View all 500 →]"
________________


2. Pattern Detection False Positive
Failure: System detects "pattern" from coincidental events (3 unrelated GPU crashes).
Safe Default:
* Show pattern with confidence level: "Possible pattern (medium confidence)"
* Include: "These events may be unrelated. Review details before acting."
* Never auto-enable mitigations (always ask first)
________________


3. Cost Data Missing
Failure: Incident occurred but cost impact not logged (cost tracking failed).
Safe Default:
* Show narrative with: "Cost impact: Unable to calculate (data unavailable)"
* Flag for manual review: "Cost data missing — add to financial audit queue"
* Emit: cost_tracking_failure event
________________


4. Weekly Digest Email Fails to Send
Failure: Email delivery service down (SendGrid outage).
Safe Default:
* Retry 3 times with exponential backoff
* If all retries fail → Store digest in FOCL: "View last week's digest (email delivery failed)"
* Alert Founder on next login: "Weekly digest email failed to send (available in FOCL)"

2. Incident Data Retention

Incidents stored indefinitely for audit trail.
User PII in incidents: Aggregated only (user counts, not names).
________________


3. Financial Data Export Security
Exported financial data:
* CSV downloads require step-up authentication (biometric / TOTP)
* Downloads logged to audit trail
* No automatic email of financial exports (manual download only)
